# Sprint 73 Plan — The Autopilot v2 (Smart Model Routing)

**Par:** 4 (4 tickets)
**Slope:** 2 (cross-cutting: loop analysis, model selector, CLI)
**Theme:** Extend model routing with per-ticket-type tracking, cost-performance analysis, and a `slope loop config recommend` CLI surface

## Context

`selectModel()` in `src/cli/loop/model-selector.ts` already supports data-driven overrides via `slope-loop/model-config.json`. That file is generated by `slope-loop/analyze-scorecards.ts` from scorecard history. The current analysis tracks success rates per-club only. S73 extends it with per-ticket-type breakdowns, cost-performance tradeoffs, and surfaces recommendations via a new CLI subcommand.

**Existing interfaces to build on:**
- `ModelConfig` in `src/cli/loop/types.ts` — `success_rates`, `cost_per_success`, `recommendations` fields
- `selectModel()` in `src/cli/loop/model-selector.ts` — reads `model-config.json` recommendations
- `slope-loop/analyze-scorecards.ts` — mines scorecard data → `analysis.json` + `model-config.json`
- `slope-loop/analysis.json` — current analysis output
- `BacklogTicket.estimated_tokens`, `.max_files`, `.club` — routing inputs

## Tickets

### T1: Model success rate tracking — per-club, per-ticket-type breakdown
**Club:** short_iron
**Files:** `slope-loop/analyze-scorecards.ts`, `src/cli/loop/types.ts`, `tests/cli/loop/model-selector.test.ts`

**Problem:** `analyze-scorecards.ts` currently aggregates success rates per-club only. We can't tell if `short_iron` performs differently on `feature` vs `hardening` tickets, or whether multi-file tickets (max_files ≥ 2) succeed more with the API model.

**Approach:**
- Extend `ModelConfig.success_rates` with a second-level breakdown: per-club-per-strategy and per-club-per-ticket-type
- Update `analyze-scorecards.ts` to compute these cross-dimensional rates from sprint result files in `slope-loop/results/`
- Add a `min_samples` threshold (≥ 3 results) before emitting a recommendation for a dimension — avoid routing decisions from single data points
- Update `ModelConfig` interface in `types.ts` if structure changes

**Hazard watch:** `ModelConfig` is consumed by `selectModel()` — any structural change must preserve backward compatibility or update `loadModelConfig()` in parallel.

### T2: Adaptive model selection — apply cross-dimensional recommendations
**Club:** long_iron
**Files:** `src/cli/loop/model-selector.ts`, `src/cli/loop/types.ts`, `tests/cli/loop/model-selector.test.ts`

**Problem:** `selectModel()` only checks `modelConfig.recommendations[club]` — a flat per-club lookup. With T1's richer data, we can factor in ticket type and strategy to make better routing calls.

**Approach:**
- Extend `selectModel()` to accept `ticketType` (feature/bugfix/chore) as an additional input
- Check cross-dimensional recommendations in priority order: club+type → club+strategy → club-only → defaults
- Only apply a cross-dimensional recommendation if it has enough samples (use `min_samples` from T1)
- If no data-driven override applies, fall through to existing club-default logic unchanged

**Hazard watch:** `selectModel()` is called from the loop runner — signature change must not break existing callers. Add the new params as optional with defaults.

### T3: Model cost-performance tradeoff — factor cost into routing
**Club:** short_iron
**Files:** `slope-loop/analyze-scorecards.ts`, `src/cli/loop/model-selector.ts`, `slope-loop/model-selector.ts`

**Problem:** `analysis.json` has `cost_per_success` but it's not used in routing decisions. A model with 80% success rate at $0.001/ticket beats a model with 85% success at $0.10/ticket for routine work.

**Approach:**
- Add a `cost_adjusted_score = success_rate / (cost_per_success + epsilon)` metric to analysis output
- In `selectModel()`, if both local and API recommendations have sufficient samples, pick the one with better cost-adjusted score rather than always deferring to API
- Emit `cost_adjusted_scores` in `model-config.json` so it's inspectable
- Update `slope-loop/model-selector.ts` (shell-script companion) to reflect the same logic

**Hazard watch:** `slope-loop/model-selector.ts` is a separate file from `src/cli/loop/model-selector.ts` — both need updating to stay in sync.

### T4: `slope loop config recommend` — surface optimal config from data
**Club:** wedge
**Files:** `src/cli/commands/loop.ts`, `tests/cli/loop/loop-cmd.test.ts`

**Problem:** Model routing recommendations live in `slope-loop/model-config.json` but there's no CLI command to read and explain them. Operators have to read raw JSON.

**Approach:**
- Add `slope loop config recommend` subcommand to `loop.ts`
- Reads `model-config.json`, formats recommendations as a table: club → recommended model → reason → sample count → success rate
- If `model-config.json` is missing or stale (generated_at > 7 days), warn and suggest running `slope loop analyze`
- Add a `--json` flag for machine-readable output

**Hazard watch:** `loop.ts` already has a `config` subcommand with `--show` and `--set` flags — `recommend` must slot in cleanly without disrupting existing subcommand routing.
