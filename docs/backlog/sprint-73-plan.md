# Sprint 73 Plan ‚Äî The Autopilot v2 (Smart Model Routing)

**Par:** 4 (4 tickets)
**Slope:** 2 (cross-cutting: loop analysis, model selector, CLI)
**Theme:** Extend model routing with per-sprint-type tracking, cost-performance analysis, and a `slope loop config recommend` CLI surface

## Context

`selectModel()` in `src/cli/loop/model-selector.ts` already supports data-driven overrides via `slope-loop/model-config.json`. That file is generated by `slope-loop/model-selector.ts` (reads JSONL result files from `slope-loop/logs/`, writes `model-config.json`). Note: `slope-loop/analyze-scorecards.ts` is a separate script that generates `analysis.json` and `backlog.json` ‚Äî it does NOT produce `model-config.json`. The current model config tracks success rates per-club only. S73 extends it with per-sprint-type breakdowns, cost-performance tradeoffs, and surfaces recommendations via a new CLI subcommand.

**Key distinction:** `BacklogSprint.type` is `'feature' | 'bugfix' | 'chore'` and `BacklogSprint.strategy` is `'hardening' | 'testing' | 'cleanup' | 'documentation' | 'roadmap'`. Both are sprint-level, not per-ticket. The new routing dimension uses these sprint-level fields.

**Existing interfaces to build on:**
- `ModelConfig` in `src/cli/loop/types.ts` ‚Äî `success_rates`, `cost_per_success`, `recommendations` fields
- `selectModel()` in `src/cli/loop/model-selector.ts` ‚Äî reads `model-config.json` recommendations
- `slope-loop/model-selector.ts` ‚Äî standalone TS script that reads JSONL logs ‚Üí writes `model-config.json` (has its own `TicketResult` interface and hardcoded model names ‚Äî diverged from CLI-side)
- `BacklogSprint.type`, `BacklogSprint.strategy` ‚Äî routing inputs available at the sprint level
- `BacklogTicket.estimated_tokens`, `.max_files`, `.club` ‚Äî routing inputs available per-ticket

## Tickets

### T1: Model success rate tracking ‚Äî per-club, per-sprint-type breakdown
**Club:** short_iron
**Files:** `slope-loop/model-selector.ts`, `src/cli/loop/types.ts`, `tests/cli/loop/model-selector.test.ts`

**Problem:** `slope-loop/model-selector.ts` currently aggregates success rates per-club only. We can't tell if `short_iron` performs differently on `feature` vs `hardening` sprints, or whether `roadmap` strategy tickets benefit more from the API model.

**Approach:**
- Extend `ModelConfig.success_rates` with a second-level breakdown: per-club-per-strategy and per-club-per-sprint-type
- Update `slope-loop/model-selector.ts` to compute these cross-dimensional rates from JSONL result files
- Add a configurable `min_samples` threshold (default ‚â• 3 results) before emitting a recommendation for a dimension ‚Äî avoid routing decisions from sparse data
- Update `ModelConfig` interface in `types.ts` if structure changes
- Fix existing inconsistency: `ModelConfig.recommendations` types `model` as `'api' | 'local'` but `slope-loop/model-selector.ts` writes plain `string` ‚Äî align both

**Hazard watch:** `ModelConfig` is consumed by `selectModel()` ‚Äî any structural change must preserve backward compatibility or update `loadModelConfig()` in parallel. Cross-dimensional breakdown will be sparse (5 clubs x 6 strategies = 30 cells) ‚Äî ensure fallback chain tests cover the common case where most cells are empty.

### T2: Adaptive model selection ‚Äî apply cross-dimensional recommendations
**Club:** long_iron
**Files:** `src/cli/loop/model-selector.ts`, `src/cli/loop/types.ts`, `tests/cli/loop/model-selector.test.ts`

**Problem:** `selectModel()` only checks `modelConfig.recommendations[club]` ‚Äî a flat per-club lookup. With T1's richer data, we can factor in sprint type and strategy to make better routing calls.

**Approach:**
- Extend `selectModel()` to accept optional `sprintType` (`BacklogSprint['type']`) and pass `strategy` (already a param) through to the recommendation lookup
- Check cross-dimensional recommendations in priority order: club+sprintType ‚Üí club+strategy ‚Üí club-only ‚Üí defaults
- Only apply a cross-dimensional recommendation if it has enough samples (use `min_samples` from T1)
- If no data-driven override applies, fall through to existing club-default logic unchanged
- Must test the full fallback chain including the common case where all cross-dimensional cells are empty (cold start)

**Hazard watch:** `selectModel()` is called from the loop runner ‚Äî signature change must not break existing callers. Add `sprintType` as optional with default. The `strategy` param already exists ‚Äî just needs to be wired into the recommendation lookup.

### T3: Model cost-performance tradeoff ‚Äî factor cost into routing
**Club:** short_iron
**Files:** `slope-loop/model-selector.ts`, `src/cli/loop/model-selector.ts`

**Problem:** `model-config.json` has `cost_per_success` but it's not used in routing decisions. A model with 80% success rate at $0.001/ticket beats a model with 85% success at $0.10/ticket for routine work.

**Approach:**
- Add a `cost_adjusted_score = success_rate / (cost_per_success + epsilon)` metric to `slope-loop/model-selector.ts` output
- In `src/cli/loop/model-selector.ts` `selectModel()`, if both local and API recommendations have sufficient samples, pick the one with better cost-adjusted score rather than always deferring to API
- Emit `cost_adjusted_scores` in `model-config.json` so it's inspectable

**Hazard watch:** `slope-loop/model-selector.ts` has diverged significantly from `src/cli/loop/model-selector.ts` ‚Äî it has its own `TicketResult` interface (lines 21-27), hardcoded model names (`ollama/qwen2.5-coder:32b`, `openrouter/minimax/minimax-m2.5`), and its own recommendation logic. This is NOT a simple companion sync ‚Äî treat it as a separate implementation that needs its own cost-adjusted logic. Consider whether to consolidate or keep separate.

### T4: `slope loop config recommend` ‚Äî surface optimal config from data
**Club:** wedge
**Files:** `src/cli/commands/loop.ts`, `tests/cli/loop/loop-cmd.test.ts`

**Problem:** Model routing recommendations live in `slope-loop/model-config.json` but there's no CLI command to read and explain them. Operators have to read raw JSON.

**Approach:**
- Add `recommend` as a positional arg to the config subcommand: `slope loop config recommend`
- Requires refactoring `configSubcommand` to check `args[1]` before falling into the existing `--show`/`--set` flags pattern (currently flat flag-based, not a subcommand router)
- Reads `model-config.json`, formats recommendations as a table: club ‚Üí recommended model ‚Üí reason ‚Üí sample count ‚Üí success rate
- If `model-config.json` is missing or stale (generated_at > 7 days), warn and suggest running `slope loop analyze`
- Add a `--json` flag for machine-readable output

**Hazard watch:** `configSubcommand` uses flat `flags` pattern (`--show`, `--set k=v`). Adding `recommend` means checking `args[1]` first, then falling through to flags. Test that existing `--show` and `--set` behavior is preserved after the refactor.
